# 07 — Состязательная проверка гэпов

Топ-14 фич roadmap проверены отдельными агентами **по реальному коду**: действительно ли гэпа ещё нет
и выполнимо ли предложение на текущем стеке (Python + Pillow + numpy + опц. ONNX).

**Итог: 14/14 — гэп реален, 14/14 — выполнимо.** Ни одна из приоритетных фич не оказалась «уже сделанной»
или нереализуемой. Ниже — где встроить, главный риск и уточнённая оценка.

## Ключевые уточнения по коду

### Прямоугольные таттеты W×H
- **Куда:** `config.py` (`NamedTarget` + `ExportConfig.targets`); `resize.py` (`render_target`); `export.py`
  (ветка `if ecfg.targets`); `pipeline.py:stage_export`; `crop.py` (переиспользовать `_offset` из `fit_square`).
- **Риск:** `Image.paste` цветного bg требует аккуратной работы с альфа-каналом; `bg='auto'` (медиана по
  краю) — считать по edge-пикселям; узкие/высокие ассеты (scale<1). Contact-sheet и web-UI должны уметь
  несквадратные превью.
- **Важно:** `fit` в `CropConfig` — фактически мёртвый код (кроме проверки `!=width`); таргеты должны жить
  в `ExportConfig`. Версионирование конфига не нужно — изменение чисто аддитивное.

### QA-предупреждения
- **Куда:** `export.py:build_export` и `pipeline.py:process_object` — собрать проверки после
  `render_sizes`, сложить в `meta['warnings']`; бейджи в `web/app.js:preview()`.
- **Важно:** канал `meta['warnings']` **уже существует** (используется для ошибки `.icns`), но наполняется
  только при ошибках — здесь нужна **проактивная** QA по numpy. Кэшировать исходный размер, чтобы не
  гонять проверки на каждый размер. Пороги параметризовать против ложных срабатываний.

### Авто-классификация типа ассета
- **Куда:** `pipeline.py:analyze()` — добавить `asset_kind/suggested_preset` в `meta` (строки 72-76);
  `server/app.py:api_analyze()` — вернуть `kind/preset/reasons`. **Чисто аддитивно, нулевой риск для
  существующего кода** — старые клиенты игнорируют новые поля.
- **Риск:** хрупкость эвристик (aspect не отличит wordmark от баннера; квантизация цветов промахивается на
  градиентах). Смягчается `confidence`-скором, списком `reasons` и тем, что UI показывает как «Похоже
  на…», а не авто-применяет.

### Saliency-центрирование
- **Куда:** `resize.py:resize_to()` + `crop.py:_offset()` + `config.py:CropConfig.focus`. Нового
  `render_target` в коде ещё нет — речь о параметризации существующих функций.
- **Риск:** регрессия центрирования квадратных кропов (`_offset` используется в `fit_square`) — нужно
  покрыть тестами; FFT медленна на 4K — по умолчанию Sobel. `focus=[fx,fy]` в нормализованных [0..1].

### Платформенные бандлы
- **Куда:** `ExportConfig` (`bundle_kind`, `safe_area_pct`); `build_export` после PNG — генераторы
  `_bundle_pwa/_bundle_android/_bundle_ios/_bundle_favicon`; пресеты.
- **Риск:** iOS `Contents.json` — 30+ размеров; Android — таблица px↔density + двуслойная adaptive-структура
  (выделение силуэта — **отдельная задача**); проверка валидности требует Xcode/Android Studio.
- **Минимизация риска:** начать с **PWA** (путь M: 1 JSON + 3 PNG + HTML, проверяется Lighthouse), затем
  iOS, затем Android. Полный путь — XL (3-4 дня).

### AVIF + анимация — самый недооценённый по объёму
- **Оценка XL**, не S. `export.py` сейчас предполагает один `Image` на размер; для анимации — `sizes ×
  frames` объектов в RAM (риск всплеска памяти на batch). APNG в Pillow экспериментален; animated WebP
  стабильнее.
- **Противоречие, требующее проверки в окружении:** один агент утверждает «Pillow **не** поддерживает
  AVIF», другой — «Pillow 12 уже установлен, `features.check('avif')==True` подтверждён». **Перед
  реализацией явно проверить `PIL.features.check('avif')` на целевой машине** и держать try-import с
  фолбэком на WebP+note (паттерн `_remove_ai`). Рекомендация: animated-WebP сначала, APNG опционально,
  GIF→анимация только по явному выбору пользователя (чекбокс), лимит кадров в квоте.

### История / проекты — есть архитектурный нюанс
- **Важно:** `/api` (server/app.py, **без auth**, in-memory сессии) и `/saas/routes` (HTML, auth
  обязателен) — разные слои. Для рабочей истории нужно либо поднять auth в `/api/upload` и привязать
  `session_id→user_id` в БД, либо сделать отдельный `/api/export_history` для auth-юзеров. Рекомендован
  путь 1 (откроет cross-device/мобильный API). Журналировать **конфиг**, а не байты результата —
  приватно и воспроизводимо.

### Device-frame mockups
- **Куда:** `build_export` как опц. постпроцессинг или стадия после `stage_export`; `ExportConfig.device_spec`.
- **Важно:** существующие `android.json`/`ios.json` — для **иконок** (48-512px), не для скриншотов сторов
  (1290×2796 и т.д.) — нужны отдельные `appstore-screenshots.json`/`playstore-screenshots.json`.
  `core/mockup.py` и `assets/` не существуют — гэп подтверждён.

### Контурная обводка — нюанс масштабирования
- **Важно:** контур одинаковой **пиксельной** толщины на всех размерах выглядит неправильно (1px на 16px
  почти не виден, на 512px — слишком тонко). Нужна **адаптивная толщина** (`outline_pct` или `max(1,
  size//96)`). Кэшировать дилатированную маску один раз, не пересчитывать на каждый размер.

### Спрайт-атлас
- **Важно:** использовать выход `detect.split_objects` (каждый bbox = кадр) → новая стадия + `ExportConfig.
  atlas=true`. Shelf-упаковка проста (~100-150 строк), но проектировать на расширяемость (guillotine/
  maxrects). Определиться: CSS-спрайты (x,y,w,h) vs game-atlas (rotation/trim). Главный объём — тесты
  (round-trip валидация координат).

## Что это значит для приоритезации

Проверка подтвердила структуру roadmap: **прямоугольные таргеты — действительно фундамент** (без них
соцсети/стор/печать/email невозможны), а самые дешёвые «умные» победы (QA-предупреждения, классификация,
JPG/retina) реализуемы за часы и почти без риска для существующего кода. Самые крупные по объёму —
**AVIF/анимация** и **полные платформенные бандлы** (особенно Android adaptive) — стоит дробить на
под-этапы и начинать с самой простой части (animated-WebP, PWA-манифест).
